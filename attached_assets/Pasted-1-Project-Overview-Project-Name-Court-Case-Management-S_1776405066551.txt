1.Project Overview


Project Name: 
• Court Case Management System 
Purpose: 
• Digitize and organize court case workflows, enabling lawyers, admins, and judges to manage cases, schedule hearings, and track status progression 
System Type: 
• REST-based backend application (API-first design) 
Core Functionality: 
• Allow lawyers to register and create cases 
• Allow admins to manage and assign cases 
• Allow judges to handle hearings and update outcomes 
• Maintain a timeline of hearings for each case 
• Track case status progression (Pending → Ongoing → Closed) 
• Ensure secure access using authentication and role-based authorization 
Out of Scope: 
• Real legal document verification, Evidence/media file storage, Payment systems, Public access portal, AI decision making

3. API END3. API ENDPOINTS SPECIFICATION
POINTS SPECIFICATION 
3.1 AUTHENTICATION ENDPOINTS 
3.1.1 Register User 
Method 	POST
Route 	/auth/register
Description 	Register a new user
Authentication 	Not Required


Request Body: 
{ "name": "string", "email": "string (unique)", "password": "string (min 8 chars)", "role": "string (lawyer|admin|judge)", "phone": "string (optional)"} 
Response (201 Created): 
{ "id": "integer", "name": "string", "email": "string", "role": "string",  "createdAt": "timestamp"} 
Error Responses: 
• 400: Missing/invalid fields | 409: Email already registered | 422: Invalid role Normal Course: 
1. User submits registration with valid data 
2. System validates email uniqueness
3. Password hashed and stored securely 
4. User record created in database 
3.1.2 Login User 
Method 	POST
Route 	/auth/login
Description 	Authenticate user and generate JWT
Authentication 	Not Required


Request Body: 
{ "email": "string", "password": "string"} 
Response (200 OK): 
{ "token": "string (JWT)", "refreshToken": "string", "user": { "id": "integer", "name": "string", "email": "string", "role": "string" }} 
Error Responses: 
• 400: Missing email/password | 401: Invalid credentials 
3.1.3 Refresh Token 
Method 	POST
Route 	/auth/refresh
Description 	Generate new access token
Authentication 	Required


Request Body: 
{ "refreshToken": "string"} 
Response (200 OK): 
{ "token": "string (JWT)"}

4. ERROR HANDLING STRUCTURE
HANDLING STRUCTURE 
The API uses standard HTTP status codes and provides structured error responses. Standard Error Response Format: 
{ "success": false, "error": { "code": "string", "message": "string",  "details": "object (optional)", "timestamp": "timestamp" }} 
HTTP Status Codes:
201	Created	Resource successfully created
400	Bad Request	Invalid request data
401	Unauthorized	Missing/invalid token
403	Forbidden	Insufficient permissions
404	Not Found	Resource not found
200	OK	Request successful
409	Conflict	Duplicate resource
422	Unprocessable	Invalid data format
429	Rate Limit	Too many requests
500	Server Error	Server encountered error
 

Common Error Scenarios: 
• Missing Token: 401 Unauthorized 
• Expired Token: 401 Unauthorized 
• Invalid Token: 401 Unauthorized 
• Insufficient Permissions: 403 Forbidden 
• Resource Not Found: 404 Not Found 
• Data Validation Failure: 400 Bad Request 
• Duplicate Resource: 409 Conflict 
• Server Error: 500 Internal Server Error 
5. MI5. MIDDLEWARE FUNCTIONALITIES
DDLEWARE FUNCTIONALITIES 
Middleware components provide cross-cutting concerns and process requests before they reach route handlers. 
5.1 Authentication Middleware 
Purpose: 
• Validates JWT tokens in request headers 
• Verifies token signature and expiration 
• Extracts user information from token claims 
• Attaches user object to request for downstream handlers 
Implementation Details: 
• Checks Authorization header format: Bearer <token> 
• Returns 401 if token missing or invalid 
• Returns 401 if token expired 
• Bypasses /auth endpoints (register, login) 
5.2 Authorization Middleware 
Purpose: 
• Enforces role-based access control (RBAC) 
• Restricts endpoints to specific user roles 
Role Definitions: 
• Lawyer: Create/view own cases, manage own profile 
• Admin: Assign cases, manage all cases, view all users 
• Judge: View assigned cases, schedule/update hearings 
Implementation Details: 
• Checks user role from authenticated request 
• Returns 403 Forbidden if role not permitted 
• Runs after authentication middleware
5.3 Request Validation Middleware 
Purpose: 
• Validates request body, query, and route parameters • Ensures data types match expected schemas 
• Sanitizes input to prevent injection attacks 
Implementation Details: 
• Validates email format using regex 
• Validates password length (minimum 8 characters) • Validates date/time formats 
• Returns 400 with detailed validation errors 
5.4 Error Handling Middleware 
Purpose: 
• Catches unhandled errors and exceptions 
• Converts errors to standardized API responses • Logs errors for monitoring and debugging 
Implementation Details: 
• Catches exceptions from route handlers 
• Distinguishes between client (4xx) and server (5xx) errors • Returns generic messages (hides implementation details) • Logs detailed info server-side 
5.5 Request Logging Middleware 
Purpose: 
• Logs all requests for audit and debugging 
Implementation Details: 
• Logs: Method, URL, user ID, IP, response code • Records timestamp and processing time 
• Does not log sensitive data (passwords, tokens) 5.6 CORS Middleware 
Purpose: 
• Handles Cross-Origin Resource Sharing headers Implementation Details: 
• Sets Access-Control-Allow-Origin header 
• Specifies allowed HTTP methods 
• Specifies allowed headers (Content-Type, Authorization) • Handles preflight OPTIONS requests 
5.7 Rate Limiting Middleware 
Purpose:
• Prevents abuse by limiting requests per user/IP Implementation Details: 
• Limits: 100 requests per 15 minutes per user • Returns 429 Too Many Requests when limit exceeded • Tracks by user ID (authenticated) or IP address 
5.8 Body Parser Middleware 
Purpose: 
• Parses incoming request bodies (JSON) 
Implementation Details: 
• Accepts application/json content type 
• Limits request size to prevent large payloads • Returns 413 Payload Too Large if exceeded
